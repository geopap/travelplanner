- **XS**: < 2 hours
- **S**: 2–8 hours

- **L**: 3–5 days

| B-008 | **Accommodations**: As a trip organizer I want to log hotel stays with check-in/check-out dates and confirmation details spanning multiple days so that lodging is visible across the full itinerary. | P1 | M | backlog | 2 | B-005 | (1) "Add accommodation" form: hotel name or place link, check-in date, check-out date, confirmation number, cost per night, total cost, notes. (4) On the check-in day view, a "Check in: [name]" indicator appears; on check-out day, a "Check out" indicator. (5) Accommodations listed on trip overview in check-in date order with name, dates, and total cost. |
| B-009 | **Google Places search proxy**: As a trip organizer I want to search for places by name and see results with addresses and coordinates so that I can link real places to my itinerary and bookmarks without leaving the app. | P1 | L | backlog | 2 | B-001 | (1) `/api/places/search?q=[query]` endpoint accepts query string; authenticates caller via session. (2) Google Places API key held server-side only — never returned to client. (3) Results checked against `places` cache table first (match on `google_place_id`); cache hit returns cached data. (4) Cache miss calls Google Places API, stores result in `places` table, returns to client. (5) Rate limit: 30 requests/minute per user. (6) Returns: `google_place_id`, name, address, lat, lng, category. (7) Search fires after 2+ characters; results appear within 800ms for cache hits. |
| B-010 | **Place detail cache and page**: As a trip organizer I want to view a rich place detail page (photos, hours, rating, website) so that I can evaluate a restaurant or sight before bookmarking it. | P1 | M | backlog | 2 | B-009 | (1) `/api/places/[googlePlaceId]` fetches full place details from Google Places API if not cached or TTL expired (7 days). (2) Full details stored in `places.cached_details` JSONB with `cached_at` timestamp. (3) Place detail page at `/places/[id]` renders from cache: name, photos (up to 5), address, rating (1–5), opening hours, website, Google Maps link. (4) Google attribution text displayed on all place data surfaces. (5) If place is not yet cached (first visit), page shows a loading state while the API call completes. |
| B-011 | **Bookmarks**: As a trip organizer I want to bookmark places (restaurants, sights, museums, shopping) for my trip so that I have a curated shortlist of places I want to visit. | P1 | M | backlog | 2 | B-010, B-004 | (1) "Bookmark for [trip name]" button on place detail page creates a `bookmarks` row: `trip_id`, `place_id`, `category`, `notes`, `user_id`. (3) Duplicate bookmark (same trip + place) is rejected with "Already bookmarked" feedback — no duplicate row created. (4) Bookmarks listed on trip Places tab grouped by category, sorted by name within group. (5) Viewer can view bookmarks but cannot add or delete. (6) Editor/owner can add and delete bookmarks. |
| B-012 | **Trip member invite and accept**: As a trip owner I want to invite someone by email to join my trip as an editor or viewer so that we can plan together. | P1 | M | backlog | 3 | B-001, B-004 | (1) Owner enters email + selects role (editor/viewer) on Members tab; submits. (3) Invitation link displayed on screen: `/invite/[token]` (email delivery deferred to Phase B). (4) Invitee visits link; if not authenticated, redirected to sign-in/sign-up with return URL. (5) On accept: token validated (exists, not expired, not used); `trip_members` row created with the invitation's role; `trip_invitations.used_at` set. (6) Invitee redirected to trip overview. (7) Expired/invalid token shows a clear error; no trip data exposed. |
| B-013 | **Member role management**: As a trip owner I want to view all trip members, change their roles, and remove members so that I control who can edit or view the trip. | P1 | S | backlog | 3 | B-012 | (1) Members tab lists all accepted members: display name or email, role badge, join date. | B-014 | **Budget and expenses**: As a trip organizer I want to set a trip budget and log categorized expenses with amount, currency, who paid, and who it is split among so that I can track spending against my budget. | P1 | M | backlog | 3 | B-004, B-012 | (1) Trip budget (optional) set on trip create or edit; stored in `base_currency`. (5) Expense list on Budget tab: date-desc order, filterable by category and paid_by. (6) Per-member totals section shows how much each member paid and their net balance based on splits. |
| B-015 | **Leaflet map — day view**: As a trip member I want to see a map of all the places on a given day so that I can visualize the route and proximity of stops. | P2 | M | backlog | 4 | B-006, B-010 | (1) Each day view includes a collapsible map section rendered with `react-leaflet`. | B-017 | **Profile management**: As a registered user I want to set my display name and avatar so that trip partners see my name rather than my email in member lists. | P2 | S | backlog | 4 | B-001 | (1) Profile page at `/settings/profile` shows current display name and avatar. | B-018 | **Session protection and route guards**: As the system I need all authenticated routes to reject unauthenticated requests so that trip data is never accessible without a valid session. | P0 | S | backlog | 1 | B-001 | (1) All routes under `/trips`, `/invite`, `/settings`, and `/api/**` (except `/api/auth/**`) require a valid Supabase session.
